## Context

家庭记忆项目已具备产品设计文档与实现计划文档，但当前缺少统一的可执行规格体系，导致需求、设计与实施之间存在断层风险。本次变更目标是把“家庭空间、提议确认、自动草稿、记录沉淀、时间线筛选、主题皮肤、治理与审计”统一抽象为可验证能力。

当前约束如下：
- 第一版定位网页端最小可用闭环，不引入评论、点赞、复杂审批与积分体系。
- 已有界面设计稿将作为视觉对照输入，但必须通过规格转化为可测试行为，而非仅凭视觉描述开发。
- 现有 `docs/superpowers/specs` 与 `docs/superpowers/plans` 提供了业务语义与实施方向，需要与 OpenSpec 工件对齐。

关键干系方包括产品、前端、后端、测试。交付成功标准是：OpenSpec 工件可直接驱动 `/opsx:apply` 进入实施，并能覆盖所有关键验收场景。

## Goals / Non-Goals

**Goals:**
- 形成完整变更工件链路：`proposal -> design -> specs -> tasks`。
- 以能力为单位定义可验证需求，覆盖提案中的全部新增能力。
- 明确界面设计稿治理规则：路径、版本、页面映射、变更同步机制。
- 建立可量化非功能基线：性能、可访问性、上传约束、审计可追踪。
- 输出可执行任务清单，并与既有计划文档中的任务节奏对齐。

**Non-Goals:**
- 不在本次变更中直接实现业务代码。
- 不引入第一版范围外能力（评论、点赞、复杂权限层级、自定义主题编辑器）。
- 不替代现有产品文档，而是将其转化为可执行规格契约。

## Decisions

### 决策 D-01：以“能力拆分”作为规格主组织方式
- 方案选择：每个能力对应一个 `specs/<capability>/spec.md`，并使用统一要求与场景格式。
- 选择原因：可追踪性最好，后续任务可精确映射到能力，验收可逐能力执行。
- 备选方案：将所有需求放在一个总规格文件中。
- 未选原因：难以维护、难以并行开发、变更影响面不清晰。

### 决策 D-02：界面设计稿采用“设计输入”而非“唯一真理”
- 方案选择：在设计文档中约定设计稿路径、版本、页面映射与同步流程；实现验收以规格行为为准、视觉以设计稿为对照。
- 选择原因：避免视觉稿变更影响业务契约稳定性，同时保证 UI 一致性。
- 备选方案：直接以设计稿文本作为需求来源。
- 未选原因：难以覆盖边界行为、异常流程与非功能要求。

### 决策 D-03：主题能力按第一版最小范围落地
- 方案选择：支持浅色/深色 + 系统跟随 + 手动切换 + 主题偏好持久化。
- 选择原因：满足实用场景并控制实现复杂度。
- 备选方案：支持多主题自定义与主题编辑器。
- 未选原因：超出 MVP 范围，且需要额外设计系统与治理流程。

### 决策 D-04：权限与审计采用“家庭空间内最小边界”
- 方案选择：默认家庭成员可见；关键写操作（确认提议、发布记录、成员邀请）记录审计字段。
- 选择原因：符合第一版“私密且协作”的产品原则，同时为排障留证据。
- 备选方案：立即引入细粒度角色权限。
- 未选原因：复杂度高，且不属于第一版必须能力。

### 决策 D-05：任务清单与既有计划双向映射
- 方案选择：OpenSpec `tasks.md` 按“基础设施 -> 核心闭环 -> 体验治理 -> 验收发布”分组，并在任务描述中标注对应计划主题。
- 选择原因：便于团队沿用既有计划语境，又能获得 OpenSpec 的执行与追踪能力。
- 备选方案：完全重写一套与原计划无关的任务。
- 未选原因：团队迁移成本高，且容易产生重复维护。

### 决策 D-06：量化阈值采用“默认值 + 可配置”策略
- 方案选择：在规格中给出第一版默认阈值（例如首屏可交互 3 秒、关键反馈 1 秒、上传上限 10MB、对比度 4.5:1），并在实施中允许通过配置调优。
- 选择原因：既保证验收可执行，又避免在不同环境下被固定常量绑定。
- 备选方案：仅写“可接受”“尽快”等定性描述。
- 未选原因：不可测试、不可验收、跨团队理解容易漂移。

### 决策 D-07：设计稿路径与主题持久化策略冻结
- 方案选择：界面设计稿统一固定在 `design/` 目录，并采用“日期-版本号”命名规则；主题偏好采用跨端同步策略，不仅在本地保存。
- 选择原因：便于设计资产检索与版本管理，且确保用户在多端访问时主题体验一致。
- 备选方案：设计稿路径灵活分散、主题仅本地持久化。
- 未选原因：会导致资产管理混乱与跨端体验割裂。

### 决策 D-08：前后端技术框架冻结（已确认）
- 方案选择：
  - 前端框架：`Nuxt 3` + `Vue 3` + `TypeScript`。
  - 后端形态：同仓接口层（`Nuxt/Nitro` 服务端接口）+ 领域服务分层（页面 -> 接口 -> 服务 -> 数据）。
  - 数据访问：`PostgreSQL` + `Drizzle`（含迁移机制）。
  - 测试体系：`Vitest`（单元测试）+ `Playwright`（端到端测试）。
  - 包管理与脚本：`pnpm`，统一脚本命名 `dev`、`build`、`test:unit`、`test:e2e`。
- 选择原因：与现有“单仓网页应用 + 十模块并行实施”的计划结构完全一致，能最大化减少联调开销与中途返工。
- 备选方案：
  - 前后端分仓（前端独立 + 独立后端服务）；
  - 传统单页应用 + 分离接口；
  - 多测试框架并行。
- 未选原因：
  - 分仓与多框架会增加第一版实施复杂度；
  - 与当前计划中文件路径与职责边界不一致；
  - 不利于统一回归与版本管理。
- 落地约束：
  - 开工前必须在计划文档中同步基线，并作为实施门禁；
  - 未经评审不得新增第二套并行框架或脚本体系；
  - 新功能必须遵循既定分层和统一测试命令；
  - 目录映射统一按 `Nuxt 3` 约定执行：页面路由使用 `pages/`（或 `src/pages/`），服务端接口使用 `server/api/`（或 `src/server/api/`）。

## Risks / Trade-offs

- [规格粒度过细导致维护成本升高] -> 通过能力边界控制，每个能力聚焦单一业务目标，避免过度拆分。
- [设计稿与规格不一致] -> 在设计中定义“变更同步检查点”，每次设计稿版本变更必须复核相关规格。
- [任务与规格漂移] -> 任务条目强制引用能力名称，评审时按“任务-规格”双向核对。
- [非功能要求难以落地测试] -> 为性能、可访问性、上传治理定义明确阈值与验收场景。
- [审计字段增加实现负担] -> 首版仅覆盖关键写操作，后续再扩展到全链路事件。

## Migration Plan

1. 完成本变更工件并通过 `openspec status --change "project-wide-spec-proposal"` 校验工件状态。
2. 按能力评审 `specs`，确认业务、前端、测试对验收口径一致。
3. 基于 `tasks.md` 进入 `/opsx:apply` 实施阶段，按任务组推进。
4. 实施中若出现范围变化，优先更新对应规格与任务，再继续开发。
5. 上线前按任务中的验收组执行回归，并保留审计与异常样例作为发布依据。

回滚策略：
- 若实施中发现需求方向不成立，可冻结变更并回退到仅保留 `proposal` 的状态，不影响现有运行系统。
- 若单项能力不可按期交付，可在 `tasks.md` 标记为后置范围，但不得破坏核心闭环能力。

## 决策引用规范

- 代码注释、任务说明、评审记录中需要引用设计决策时，统一使用 `D-xx` 编号（例如 `D-03`）。
- 一个实现点若受多个决策约束，按“主决策在前、次决策在后”写法标注，例如：`遵循 D-02, D-03`。
- 前端视觉与主题相关实现默认至少引用 `D-02` 或 `D-03`；权限与审计相关实现默认至少引用 `D-04`。
